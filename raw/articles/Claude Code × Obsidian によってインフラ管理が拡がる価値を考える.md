## 🛠️ 動作環境

| ツール | 役割 |
| --- | --- |
| Claude Code | AIエージェント（VSCode拡張として動作） |
| VSCode の統合ターミナル | SSH作業・監視の実行環境 |
| Obsidian | ナレッジ管理・Canvas・手順書 |

> **💡 なぜVSCodeか**  
> Ghostty等の単体ターミナルでも動作しますが、VSCodeにはAIエージェントが提示したコマンドを **ワンクリックでコピーできる** ボタンが存在します。  
> Ghosttyでは長いコマンドが折り返され、コピー時に不要なスペースが混入してエラーになるケースがありました。  
> 「コマンドを安全に貼り付ける」という監視体制の目的において、この差は無視できませんでした。

> **💡 なぜObsidianか**  
> Notionなどのクラウドサービスと異なり、ObsidianはすべてのデータがローカルのMarkdownファイルとして存在します。  
> これにより、MCPツールから直接ファイルを読み書きできるため、AIエージェントとの親和性が非常に高い。  
> 手順書・進捗管理といったナレッジをAIエージェントが参照・更新できることが前提となっています。  
> さらに、Canvas機能（`.canvas` ファイルはJSON形式）もAIエージェントが直接編集でき、インフラ俯瞰図を会話の流れで更新できます。  
> メモアプリケーションという性質上、AIエージェントが生成・上書きするファイルが多少雑な扱いになっても運用上は問題になりにくいです。  
> またテキストベースのため1ファイルあたり数KB程度で済み、ローカルディスクへの影響もほぼありません。

## 📌 はじめに

日々のサーバー作業で、手順書があってもミスが起きる場合もあれば、予想外のエラーが発生し慌てるケースもあると思います。  
私自身も施策の一環でサーバーに中〜大規模の設定変更を投入する機会がありますが、  
作業手順書通りに作業を行なっても意図しない動作となり、とても慌てたことがありました。

そこで、せっかくClaude Codeを使用しているなら、AIエージェントに作業を見守ってもらえないかと考えました。  
Claude Code の MCP (Model Context Protocol) を使ってリアルタイムでSSH操作を監視する仕組みを自作しました。

監視という中核ができると、同じ基盤（Claude Code + Obsidian）から自然に広がりが生まれました。  
**手順書の自動生成・AWSリソースの確認・インフラ俯瞰図の作図** ——それぞれが監視と密接に繋がっています。

この記事では、監視プロトコルの設計と進化、そしてそこから広がった活用について共有します。

## 🎯 背景・課題

### きっかけ：SSL証明書移行プロジェクト

今回の監視プロトコル設計のきっかけは、既存のSSL証明書をLet's Encryptへ移行するプロジェクトでした。

- **対象**: 50台以上のサーバー
- **作業内容**: 同じ手順を繰り返すが、サーバーごとに環境（OSや作成時代）が微妙に異なる
- **課題**: 手順書を作っても、人間がミスする（コマンドのコピペミス、手順飛ばし等）

ただし、これは証明書移行に限った話ではありません。 **複数台のサーバーに対して同じ手順を繰り返す作業全般に共通する課題** です。  
ミドルウェアの設定変更、OS更新、証明書切り替え——いずれも同じ構造を持っています。

### 解決したかったこと

**AIエージェントに実行した内容を把握してもらい、解決策を提示してもらえばイージーミスは減るのではと考えた。**  
しかし、AIエージェントにどうやってリアルタイムで監視させるか？  
誤認識や誤検知、判断に時間がかかっては意味がないため、これらのバランスをどうするか？

## 🔨 監視という中核を確立する

### v1.0：まず作ってみた

「AI エージェントに SSH 操作を把握させる」という方向性は決まった。最初のアプローチはシンプルだった。

SSH セッション中のすべての入出力を `.live` ファイルにリアルタイムで書き込み、Claude Code の MCP（Model Context Protocol）経由でそのファイルを参照させる。

```
作業ログ/2026-03-17_stg-gtserver.live
```

v1.0 で提供した MCP ツールは2つ：

| ツール | 役割 |
| --- | --- |
| `watch_sessions` | `.live` ファイルの出現を監視・セッション開始を検知 |
| `read_live_session` | セッションログ全体を取得・AI エージェントが文脈を把握 |

これで「AI エージェントがセッションログを読める」状態にはなった。しかし、実際に使ってみると別の課題が見えてきた。

---

### v1.0 を使う中で見えてきた課題

次の課題は「 **いつ AI エージェントがログを読みに行くか** 」だった。

#### ❌ 案1: ポーリング方式

v1.0 では AI エージェントが定期的に `read_live_session` を呼ぶ設計を試みた。

**問題点**:

- **ポーリング間隔が長いと、タイムディレイが大きくなる**
	- 例: 10秒ごとにチェック → 最悪10秒のラグ
		- **遠投でのキャッチボールのみになってしまう**
- **ポーリング直後にコマンドが送信されたら？**
	- 次のポーリングまで検知されない
- 頻繁にポーリングすると、無駄な API 呼び出しが増える

#### ❌ 案2: ユーザーに「実行しました」と報告させる

**考え方**: コマンド実行後、ユーザーが「実行した」と Claude Code に入力する

**問題点**:

- 手間が増える（本末転倒）
- ユーザーが報告を忘れたら止まる
- リアルタイム性がない

どちらも「AI エージェントが能動的に検知する」という目標には届かなかった。

---

### v2.0 への転換：インフラエンジニアの『癖』に着目する

#### 発見した行動パターン

**rootユーザー等で空のプロンプトを複数表示（空Enter）を行うのはインフラエンジニアであれば「癖」に近い**

**具体例**:

```
[root@server]#
[root@server]# ← 空Enter
[root@server]# ← 空Enter
[root@server]# rpm -q ca-certificates
```

#### なぜこの癖が生まれるのか？

- 前の出力との区切りを作り、プロンプトを見やすい位置に整える
- コマンド実行前の「間」を作る（確認のため）

#### ⚡ この癖を v2.0 の設計に取り込む

**設計方針**:

- **空Enter が実行されたとき = AI エージェントが反応するタイミング**
- プロンプトパターン（ `$` / `#` / `>` ）の再表示を検知して `prompt_only` として返す

**利点**:

- ユーザーは普段通りの操作（特別な報告不要）
- **ポーリング不要でリアルタイムに反応**

---

### v2.0 の実装：wait\_for\_command の追加

この設計思想を実現するため、v1.0 に新しい MCP ツール `wait_for_command` を追加した。

`.live` ファイルへの書き込みを監視し、プロンプトパターンを検知した時点で以下のいずれかを返す：

| 返り値 | 意味 |
| --- | --- |
| `completed` | コマンドが実行され、出力とともに完了した |
| `prompt_only` | プロンプトが再表示されただけ（空Enter） |

`prompt_only` を受け取った場合、AI エージェントはコマンドを再提示せず即座に次の `wait_for_command` を呼ぶ。  
※これは、ユーザーがコマンドを貼り付け中の可能性があることから。

v2.0 完成時点での MCP ツール一覧：

| ツール | 役割 |
| --- | --- |
| `watch_sessions` | `.live` ファイルの出現を監視・セッション開始を検知 |
| `wait_for_command` | コマンド実行完了（または空Enter）を検知して返す |
| `read_live_session` | セッションログ全体を取得（文脈喪失時のリカバリー用） |

## 📊 中核から広がる①：SSH作業が円滑になる

### 実例1: DEVサーバー での Let's Encrypt 移行（2026-03-17）

**発生したエラー**:

- PHP cURL SSL証明書エラー（error 60: SSL certificate problem: unable to get local issuer certificate）
- Apache 2.4.6 では `SSLCertificateChainFile` の明示的指定が必須

**監視がなかったら**:

- 50台すべてで同じエラーが発生していた可能性
- 手順書の不備に気づかず、他サーバーでも同じミスを繰り返した可能性があった

**監視があったから**:

- AIエージェントが即座にエラーを検知し、手順書を修正（全手順書に `SSLCertificateChainFile` を追加）
- 他サーバーでの同じエラーを予防

### 実例2: STGサーバー01 の事前レビュー（2026-03-24）

**発見した「予想外の正常値」**:

- `HTTP/2 401 Unauthorized` （STG環境の認証設定）
- CDN経由のSectigo証明書（直接の接続時はGlobalSign）

**監視がなかったら**:

- この返り値は正常ですか？というやりとりが発生する可能性があった
- 上記の可能性は実際に確認して仕様なのか、ではないのかという確認が入ることになる

**監視があったから**:

- 手順書に予想値を事前記載し、翌日の作業で「エラー？」と慌てない
- 期待値との不一致を即座に判断可能

### 工夫：監視を安定させるための設計

#### 工夫1: 監視ループの継続

```
wait_for_command を呼ぶ
→ 結果を受け取る
→ ユーザーに報告
→ 即座に次の wait_for_command を呼ぶ（ユーザー入力を待たない）
```

**なぜ待たないのか？**:  
ユーザーは次のコマンドを貼り付けるだけなので、AIエージェントの応答待ちは遅延にしかならない

#### 工夫2: 文脈喪失への対策 — 監視中の最大リスク

会話履歴が圧縮されると、AIエージェントは「今どのサーバーで」「どの手順書の何ステップ目を実行中か」を失う。  
SSH監視体制下でこれが起きると、 **誤ったステップのコマンドを提示してしまう可能性がある** 。

**対策として設けたリカバリー機構**:

```
.live ファイル検知（現在サーバー接続中であることを確認）
→ current_task 読み込み（作業目的・ホスト名）
→ 手順書を検索・読み込み
→ ログから現在のステップを判断
```

そのため、長時間作業の前には `/compact` で事前に圧縮しておく運用ルールを設けている。

#### 工夫3: 手順書の書き方

監視と手順書は一体で機能する。手順書の書き方次第で検知精度が変わる。

- **1ステップ1コマンド** （検知精度向上）
- **`&&` は真に必要な場合のみ** （独立した操作は分割）
- **期待値を明記** （特に「正常だが予想外の値」）

## 📄 中核から広がる②：手順書作成が円滑になる

監視プロトコルで蓄積されたセッションログは、そのまま手順書の材料になる。

```
サーバー作業（監視体制下）
  ↓
.live ファイルにログが蓄積
  ↓
Claude がログを解析
  → 失敗・試行錯誤を除外
  → 成功したコマンドのみ抽出
  ↓
Obsidian の手順書テンプレートに沿って自動生成
  → ヘッダー / 作業概要 / ステップ / 切り戻し手順
```

**手動で書くと抜けがちな「切り戻し手順」まで自動で含まれる** のが特に効果的だった。

## 🗺️ 中核から広がる③：作図が円滑になる

監視によって「サーバーの実態を正確に把握する」という習慣が生まれた。  
同じ発想を AWS リソースの確認に向けると、俯瞰図の問題が解決できることに気づいた。

### 課題: インフラ俯瞰図が常に古い

複数プロジェクトを抱えるインフラチームでは、 **AWSリソースの実態と俯瞰図がズレる** という問題があった。

- CloudWatch アラームを追加したが図には反映されていない
- WAF・ACM・SSM の記載が漏れている
- 現行構成と移行後構成が混在して混乱する

### 解決策: AWSリソース確認 → 即座に俯瞰図へ反映

```
AWS CLI（describe / list / get）または terraform のコード解析
  ↓
Claude が結果を解析
  ↓
Obsidian Canvas（.canvas ファイル）を直接編集
  ↓
CloudWatch / SNS / WAF / ACM / SSM が俯瞰図に反映
```

**CLAUDE.md に AWS CLI 参照系を auto-approve 登録** しておくことで、調査からCanvas更新までが会話の流れで完結する。

```
"Bash(aws * describe*)",
"Bash(aws * list*)",
"Bash(aws * get*)"
```

※私の場合は、AWS CLI及びGitに関しては、変更(modify) や削除(delete)を伴うコマンドの実行は、例えユーザーが許可しても実行できないようCLAUDE.mdに明記しています。

### Obsidian Canvas の構造化: レイヤー色分け

俯瞰図が複雑化してきたため、 **レイヤーごとに色分けと大枠グループを統一** した。

| 色 | 用途 |
| --- | --- |
| 赤 | ネットワーク層（CF / WAF / ALB 等） |
| 黄 | CI/CD・デプロイフロー |
| 緑 | インフラ層（EC2 / ECS / Aurora 等） |
| 青 | 監視層（CloudWatch / SNS / 通知先） |
| 紫 | TLS証明書・特記事項 |

以下はaws cliにてステータス確認を行なったついでに、AIエージェントに作図が可能かを相談した結果、生成されたものです。  
作成された当初は改行コードが誤っていたり、枠組みのサイズ感が正確ではないため文字が隠れていたりと調整が必要でした。  
当初はMermaidにて作図をお願いしていましたが、生成された内容が手動変更できないことや、座標修正による位置関係の調整が困難であることから、canvasを採用しています。  
canvasは手動調整が可能であることから、大きさの変更や導線の変更等はこちらで調整し、AIエージェントには大きな枠組みでの作成をお願いしました。

![](https://storage.googleapis.com/zenn-user-upload/f87d52a72138-20260403.png)

ノードの色だけでなく、 **矢印（エッジ）にも同じ色を付与** することで、どの層のフローなのかが視覚的に一目でわかるようになりました。

さらに **CI/CD（GitHub等）は右縦列に統一配置** するレイアウトルールを設けることで、複数プロジェクトの図を並べても迷わなくなりました。

## 📋 運用での使い分け

| 状況 | 手法 | 理由 |
| --- | --- | --- |
| 短時間・単純作業 | 手動対話 | スピード優先 |
| 長時間・複雑・高リスク | 監視体制 | 安全性優先 |
| 同じ作業の2回目以降 | 手順書化 → 監視 | 再現性確保 |
| 探索フェーズ（初回） | 手動対話 | 柔軟性優先 |

## 📝 まとめ

### 設計の核心

監視（把握）を中核に置くことで、各活動が密接に繋がった。

![](https://storage.googleapis.com/zenn-user-upload/250ab17b7afa-20260403.png)

> 色分けは俯瞰図と同じルール（赤：SSH/ネットワーク、青：手順書/監視層、黄：作図/CI/CD、紫：進捗管理）に準拠しています。

1. **エンジニアの「空Enter」の癖を `prompt_only` シグナルとして活用**
2. **遅延なしのリアルタイム監視を実現（v1.0 → v2.0）**
3. **監視という中核から、手順書・作図・記録が自然に広がった**

### 得られた価値

- OSや設定がサーバーごとに異なる作業でも、判断を維持しながら迅速に進められる
- 手順書の不備を早期発見
- AWSリソースの実態と俯瞰図のズレが解消
- ログから手順書が自動生成されるため、文書化の負担がほぼゼロ
